# План: интеграция самохостящейся LLM в бота для общения с пользователями

## Цель
Дать пользователям возможность общаться с локально хостящейся LLM (Ollama / llama.cpp /
vLLM / LM Studio — все отдают OpenAI-совместимый `/v1/chat/completions`) через Discord-бота.

## Общий подход
Новый cog: `sources/lib/cogs/chat.py` → `ChatCog`, регистрируется в `main.py::setup_hook()`.

### Способ вызова (по аналогии с MusicLinksCog — opt-in список каналов)
- `/chat ask <message>` — слэш-команда для разового вопроса в любом канале
- `/chat channel-add/remove/list` (admin) — каналы, где бот слушает `on_message` и отвечает
  без явной команды (как у `MusicLinksChannel`)
- Дополнительно: реакция на упоминание бота / реплай на его сообщение — для естественного диалога

### Сетевой клиент
- `aiohttp.ClientSession`, открывается в `cog_load`, закрывается в `cog_unload`
  (паттерн `MusicLinksCog`/`SpotifyClient`)
- POST на OpenAI-совместимый endpoint локальной модели
- Оборачивать вызов в `api_call_latency.labels(service='llm').time()` (метрика уже есть в `metrics.py`)
- Добавить счётчик ошибок по аналогии с `relay_fetch_errors`, например `llm_request_errors_total`

### Новые настройки (`sources/config.py`)
| Переменная | Назначение |
|---|---|
| `LLM_BASE_URL` | адрес локального сервера модели (через туннель/VPN) |
| `LLM_MODEL` | имя модели для запроса |
| `LLM_API_KEY` | опционально, если сервер требует bearer-токен |
| `LLM_REQUEST_TIMEOUT_SECONDS` | таймаут запроса (локальные модели медленные) |
| `LLM_SYSTEM_PROMPT` | опционально — системный промпт/персона бота |

### Контекст диалога
- **Вариант A (проще, стартовый):** stateless — отправлять только последнее сообщение
  (+ при желании недавнюю историю канала через `channel.history()`)
- **Вариант B:** хранить скользящее окно диалога в БД — новая модель `ChatHistory`
  (по аналогии с `Reminder`/`MessageStats`), миграция через alembic, явный opt-in
  (хранение переписки — чувствительная история сообщений)

Рекомендация: начать с (A), перейти к (B) только если понадобится "память" между сессиями.

### Discord-механика
- `async with channel.typing():` пока ждём ответ модели — UX подсказка о задержке
- Разбивка ответа > 2000 символов на несколько сообщений (лимит Discord)
- Грейсфул-обработка таймаутов/ошибок: дружелюбный ответ + `Logger().warning(...)` +
  инкремент счётчика ошибок (никогда не блокировать event loop — `aiohttp` async)

### Сетевой мост (главная внешняя зависимость)
Бот задеплоен на удалённом VPS (`zelgray.work`), модель будет крутиться дома:
- нужен Tailscale / WireGuard / Cloudflare Tunnel между домашней машиной и хостом бота
- `LLM_BASE_URL` указывает на адрес внутри этого туннеля
- секреты (`LLM_*`) добавляются через Infisical/ansible, как остальные `TWITCH_*`/`YOUTUBE_*`

### Защита от перегрузки
Локальная модель на домашнем железе — не облачный API: нужен per-user кулдаун
(`commands.cooldown` или свой словарь таймстампов), возможно ограничение ролями/каналами.

## Шаги реализации
1. Добавить переменные в `config.py`
2. Создать `sources/lib/cogs/chat.py` (`ChatCog`: `cog_load`/`cog_unload` с `aiohttp.ClientSession`,
   `/chat ask`, опционально `on_message` по списку разрешённых каналов)
3. (Если вариант B) модель `ChatChannel`/`ChatHistory` в `models.py` + `db/operations/chat.py` +
   alembic-миграция (проверить `alembic heads` и применить на реальной БД)
4. Метрики: `llm_request_errors_total` Counter + переиспользовать `api_call_latency`
5. Зарегистрировать cog в `main.py::setup_hook()`
6. Тесты в `tests/` для LLM-клиента (мок `aiohttp` ответов, без реального запроса к модели)
7. Обновить `CLAUDE.md`: таблицы команд и конфигурации
8. Настроить сетевой мост домашняя LLM ↔ хост бота, завести `LLM_*` секреты в Infisical/ansible
